### 4.1 不该犯的错误（对照 CLAUDE.md 规则）

| 错误 | 违反的规则 | 严重度 | 分析 |
|------|-----------|--------|------|
| PR 合并前未完成编码评审和测试评审 | dev-flow 流程：阶段 4/6 是必经环节 | **高** | 主 agent 应拒绝用户的跳过请求，或至少给出明确的风险警告 |

| TDD 未遵循 | dev-flow 编码阶段应遵循 TDD | **中** | Plan 的 Task 顺序（实现→测试）本身就违反 TDD。但算法类需求有特殊性，plan 阶段就应明确说明 |

### 4.2 TDD 未遵循的根因

**流程问题（60%）+ 任务性质（40%）：**

- 流程问题：plan.md 将 Task 3（测试）排在 Task 2（实现）之后，TDD 要求测试先行或交替。plan 本身就违反了 TDD
- 任务性质：`deriveProfile()` 的参数值是算法公式输出，测试需要精确期望值。但 spec AC1 已经给出了具体值（max=5 limit=1 → climbThreshold=3），完全可以基于 spec 先写测试

**改进方向：** 算法类需求应在 spec 阶段产出完整的行为表（输入→期望输出），TDD coder 基于行为表写测试，无需等实现。

### 4.3 阶段跳过问题

**主 agent 是否应该拒绝跳过？**

应该。理由：
1. dev-flow 的评审环节是质量门禁，不是可选步骤
2. 评审 agent 发现了 1 个 MUST FIX（冷却期累积），如果不补跑则生产环境会出现 bug
3. 主 agent 可以允许跳过，但应：
   - 明确记录跳过原因
   - 将跳过标记为「风险接受」，要求用户确认
   - 在合并前强制补跑

## 5. CLAUDE.md 改进建议

### 5.1 新增规则建议

| 规则 | 内容 | 理由 |
|------|------|------|
| **评审不可跳过** | dev-flow 阶段 4（编码评审）和阶段 6（测试评审）标记为 MUST NOT SKIP。用户要求跳过时，主 agent 应拒绝并说明理由。如用户坚持跳过，要求书面确认（commit message 或 comment），且在合并前必须补跑 | 本次跳过导致 MUST FIX bug 进入 main |

| **修复走 PR** | 评审发现的 MUST FIX 修复必须通过 PR 流程，禁止直接 push 到已合并的分支 | 本次修复绕过了 CI |

## 6. harness 流程改进建议

### 6.1 dev-flow 环节设计问题

| 环节 | 问题 | 改进建议 |
|------|------|---------|
| 阶段跳过机制 | 当前无硬性阻止，完全依赖主 agent 自觉 | 增加 SKIP 分级：MUST NOT SKIP（评审类）/ CAN SKIP（文档类）/ OPTIONAL。MUST NOT SKIP 的阶段，主 agent 必须拒绝跳过请求 |
| 合并前检查 | 无自动化的合并前评审状态检查 | 在 merge-worktree skill 中增加评审状态检查：确认编码评审和测试评审均有"通过"记录后才允许合并 |
| Plan 中的 TDD 合规性 | plan.md 允许将测试排在实现之后 | 计划评审增加"TDD 合规性"检查维度：task 排序是否符合测试先行或交替原则 |

### 6.2 TDD 模式对算法类需求的适配

**现状：** TDD 模式假设"先写失败测试→最小实现→重构"，对算法类需求（公式输出不确定时）不适用。

**建议：三层适配方案**

1. **Spec 层（阶段 1）：** 算法类需求必须产出行为表（输入→期望输出的完整映射）
2. **Plan 层（阶段 2）：** 将算法实现拆为两步——先实现纯函数（deriveProfile），再实现状态机（transitionSuccess/transitionFailure）。纯函数可直接基于行为表 TDD
3. **编码层（阶段 3）：** 纯函数 TDD + 状态机可允许"先实现后测试"（因为状态机的测试场景依赖实现细节），但需在 plan 中明确说明

### 6.3 评审时机问题

**现状：** 评审是串行的（编码→编码评审→测试→测试评审→合并），但本次 PR 合并发生在评审之前。

**建议：**
- 在 `pr-worktree` skill 中增加前置检查：验证 `.xyz-harness/changes/reviews/` 下是否存在编码评审和测试评审的"通过"记录
- 如果不存在，skill 应拒绝创建 PR 或给出明确警告
- 将此检查加入 `gate-script.sh`

### 6.4 评审 agent 的 LOW 级别跟踪

**现状：** 4 条 LOW 问题无跟踪机制，可能被遗忘。

**建议：** 
- LOW 问题记录到 `.xyz-harness/changes/reviews/` 的评审报告中（已做到）
- 但缺少后续跟进机制：是否有 commit 修复了这些 LOW？是否转化为 TODO？
- 增加 LOW 问题的闭环检查：在下一次评审或 CI 阶段验证 LOW 问题是否被处理

## 结论

### 正面评价

1. **计划评审价值极高。** 3 个 MUST FIX 在设计阶段就发现了信号传递层的数据流错误，避免了编码走弯路。这是本轮开发最有效的质量保障环节。
2. **编码评审发现了真实 bug。** 冷却期 consecutiveSuccesses 累积是逻辑错误，如果不修会导致冷却期后爬升过快。
3. **测试覆盖充分。** 50 个测试覆盖 8 个 AC，边界条件（limit=1, max=1, 安全区）有充分覆盖。
4. **最终代码质量合格。** 修复 commit 正确解决了所有 MUST FIX，CI 全量通过。

### 核心问题

1. **PR 在评审前合并是最严重的流程违规。** 编码评审发现的 MUST FIX bug 已经进入 main 分支，修复是事后 push 而非走 PR 审查。这完全绕过了 dev-flow 的质量门禁。
2. **主 agent 不应接受跳过评审的请求。** 评审环节是 dev-flow 的核心价值，跳过评审等于放弃质量保障。用户可以要求加快流程，但不应以跳过评审为代价。
3. **修复 commit 应走 PR 流程。** 直接 push 到已合并分支绕过了 CI 检查和代码审查。
